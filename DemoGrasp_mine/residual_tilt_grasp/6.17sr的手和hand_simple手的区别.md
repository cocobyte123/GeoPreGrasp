# 6.17 Shadow Hand 对齐与可视化问题总结

本文记录 `shadow_simple` 与 `sr_shadow_hand` 在 replay / training 中暴露出的关键问题，后续改训练环境时应优先对照这些点。

## 1. CPU/GPU Pipeline 问题

`play_hand_only.py` 默认使用 CPU pipeline：

```text
sp.physx.use_gpu = False
sp.use_gpu_pipeline = False
```

它能稳定运行的一个原因是：Isaac Gym tensor 和脚本里新建的 PyTorch tensor 都在 CPU 上。

`rotate_play.py` 一旦改成 GPU pipeline，`gymtorch.wrap_tensor(...)` 返回 CUDA tensor，但脚本中大量 `torch.zeros(...)`、`torch.arange(...)`、`torch.tensor(...)` 默认仍在 CPU 上，会触发 device mismatch。

当前处理：

- `rotate_play.py` 默认回到 CPU pipeline，和 `play_hand_only.py` 一致。
- 新增 `--gpu_pipeline`，只有显式传入时才启用 GPU pipeline。
- GPU pipeline 下，运行态 tensor 应跟随 `root_state.device`。

后续训练环境注意：

- 训练默认是 GPU pipeline，所有新建 tensor 必须使用 `device=self.device` 或跟随已有 sim tensor。
- 尤其检查 reset、indexed set、trajectory buffer、actor id、object id、reference tensor。

## 2. Viewer 可视化崩溃问题

现象：

```text
CUDA_VISIBLE_DEVICES=1 ... headless=False
Physics Device: cuda:0
GPU Pipeline: enabled
Segmentation fault (core dumped)
```

原因判断：

- `CUDA_VISIBLE_DEVICES=1` 后，CUDA 里的物理 GPU 1 被映射成逻辑 `cuda:0`。
- Isaac Gym viewer / Vulkan 的 `graphics_device_id=0` 很可能仍按物理 GPU 0 解释。
- 物理 GPU 不一致时，viewer 初始化阶段容易 segfault。

当前处理：

- 在 `train_tilted_hand_only.py` 中加入非 headless 的自动重启逻辑。
- 当检测到 `CUDA_VISIBLE_DEVICES` 是单个非 0 GPU 时，去掉 mask，并设置：

```text
sim_device=cuda:<physical_gpu>
rl_device=cuda:<physical_gpu>
graphics_device_id=<physical_gpu>
```

日志应出现：

```text
Restarting viewer process with consistent physical GPU ordinals
Viewer devices: CUDA_VISIBLE_DEVICES=<unset>, sim=cuda:N, rl=cuda:N, Vulkan graphics_device_id=N
```

建议：

- 训练大规模用 `headless=True`。
- 可视化单独小规模运行，例如 `num_envs=16`。
- 不建议 `num_envs=7000 headless=False`，即使不崩也会很卡。

## 3. `shadow_simple` 和 `sr_shadow_hand` 不是简单包含关系

表面理解：

```text
sr_shadow_hand = shadow_simple + wrist joints
```

这个理解不完全成立。URDF frame 定义不同。

`shadow_simple`：

- base joints 最后接到 `wrist`
- `wrist -> rh_palm` 是 fixed joint
- fixed joint 有 `rpy="0 0 1.570796327"`
- 也就是 palm frame 自带一个绕 Z 的 90 度固定旋转

`sr_shadow_hand`：

- base joints 接到 `forearm`
- `forearm -> WRJ2 -> wrist -> WRJ1 -> palm`
- `WRJ2` origin 约为 `[0, -0.010, 0.213]`
- `WRJ1` origin 约为 `[0, 0, 0.034]`
- base 控制的是 forearm frame，不是 palm / wrist frame

因此：

- 两个模型的 base pose 含义不同。
- sr 多出来的不只是两个 wrist DOF，还改变了 base-to-palm 的 frame 定义。
- 不能直接把同一组 hard-coded `default_dof_pos` 当成同一只手的同一姿态。

## 4. sr 初始化位置偏前问题

现象：

- `sr_shadow_hand` 初始化时手明显比 `shadow_simple` 靠前。
- 进入 reaching / grasp 后，两者轨迹又能对齐。

根因：

- sr 的 base 定位在 forearm。
- palm 相对 forearm 有偏移：

```python
palm_offset_from_forearm = (0.0, -0.01, 0.247)
```

- tracking 阶段已经减了这个 offset：

```python
desired_pos = desired_pos - offset_world
```

- 但初始化 `default_dof_pos` 之前没有扣这个 offset。

当前处理：

- 初始化 `default_dof_pos` 时，对有 `palm_offset_from_forearm` 的 hand，将 offset 按当前 base 姿态转到 world frame，然后从 base XYZ 中扣掉。
- 这样初始 palm 位置和后续 tracking 使用同一套定义。

训练环境需要同步检查：

- `reset_idx` 中如果设置的是 base / eef / palm 初始 pose，必须明确它是 forearm、wrist 还是 palm。
- 对 sr，如果控制目标是 palm，需要在 reset 和 tracking 中都扣除 forearm-to-palm offset。
- 不能只在 tracking 中修正，否则 reset 可视化和接触初始状态会不一致。

## 5. sr 初始化旋转 90 度问题

现象：

- sr 初始化时，手面和桌面约差 90 度。
- tracking / grasp 后姿态变得更接近 `shadow_simple`。
- 直接把 `wrist_quat_offset=Rz(90deg)` 应用到初始化 base 后，初始视觉问题缓解，但成功率下降，且 reaching 中手腕会斜向下约 45 度。

重要结论：

- `wrist_quat_offset` 是轨迹 frame 到 sr palm/wrist frame 的修正。
- 它不应简单硬乘到 hard-coded default base 姿态上。

原因：

默认 sr base 角大致是：

```text
[0, 1.57, 0]
```

如果再乘 `Rz(90deg)`，会接近：

```text
[0deg, 90deg, 90deg]
```

这会落入串联欧拉/base joint 的奇异区域，reaching 插值可能走出不自然姿态，导致手腕斜向下和成功率下降。

当前处理：

- 不再把 `wrist_quat_offset` 直接注入默认 base 姿态。
- 对 `sr_shadow_hand` 新增：

```python
align_default_base_to_ref = True
```

- 初始化 base rotation 直接使用 reference frame 0 已经计算好的 `ref_base_angles[0, 0]`。

这样做的目的：

- 初始化和 tracking 使用同一个姿态来源。
- 避免 hard-coded default pose 和 trajectory pose 之间突然差 90 度。
- 避免直接乘 `Rz(90deg)` 导致欧拉奇异和 reaching 姿态异常。

训练环境需要同步检查：

- sr 的 reset 初始 base rotation 不应继续使用固定 `[0, 1.57, 0]` 作为等价手面姿态。
- 如果训练中使用 reference trajectory，reset / pregrasp 初始姿态最好从 reference frame 0 或统一的 canonical pose 生成。
- 如果必须使用 hand YAML 里的 `default_dof_pos`，要明确它是 forearm pose，不一定是 palm-aligned pose。

## 6. Trajectory Interpolation 相关判断

当前看到的成功率下降和手腕斜向下，不优先认为是插值算法本身坏了。

更可能的链路是：

```text
错误 default base orientation
-> reaching_start 姿态错误
-> slerp 从错误起点插到正确轨迹目标
-> 中间姿态斜向下
-> 接触时手面不平行桌面
-> 成功率下降
```

因此后续排查顺序建议：

1. 先确认 reset / reaching_start 的 frame 是否正确。
2. 再确认 `target_quat = ref_wrist_quat[:, 0]` 是否符合实际 palm pose。
3. 最后再看 `quat_slerp` 或 `quat_to_base_angles`。

不要优先重写插值，否则可能掩盖 frame 定义问题。

## 7. 后续训练代码需要重点同步的文件

优先检查：

- `residual_tilt_grasp/tilted_hand_only_grasp.py`
- `tasks/grasp.py`
- `tasks/hand/sr_shadow_hand.yaml`
- `tasks/hand/shadow_simple.yaml`
- `residual_tilt_grasp/train_tilted_hand_only.py`

检查项：

- reset 初始姿态是否使用 hand YAML hard-coded default。
- eef link / palm link / forearm link 的含义是否混用。
- reference trajectory 的 wrist quaternion 是否经过相同 `wrist_quat_offset`。
- sr 的 WRJ2 / WRJ1 是否被当作普通 hand DOF 训练，但 base pose 又假定在 palm 上。
- observation 里的 `eefpose` 对 sr 是 forearm、wrist 还是 palm。
- reward / success / hand-object distance 是否使用同一个 link frame。

## 8. 当前建议原则

- 对 `shadow_simple`，保持原逻辑，避免破坏已有成功率。
- 对 `sr_shadow_hand`，所有涉及“手腕/手掌初始位姿”的地方都必须显式说明 frame。
- 不要把 `wrist_quat_offset` 当作通用 reset 姿态修正。
- 位置修正使用 `palm_offset_from_forearm` 是合理的，但必须在 reset 和 tracking 中一致。
- 姿态修正更推荐从 reference frame 0 或 canonical palm pose 生成，而不是在 default pose 上硬乘固定 90 度。
- 可视化先小规模验证姿态，再跑大规模训练。

## 9. 新发现：碰撞精度与 `ssr_shadow_hand` 方案

当前更合理的方向不是继续加粗 `sr_shadow_hand` 的 box collision，而是考虑构造一只新手：

```text
ssr_shadow_hand = shadow_simple palm/fingers/collision + sr_shadow_hand wrist DOFs
```

### 9.1 URDF 结构对比

`shadow_simple`：

- URDF: `assets/shadow_hand_simple/right_with_base.urdf`
- link 数量: 37
- joint 数量: 36
- non-fixed DOF: 28
- fixed joint: 8
- collision: 21 个 mesh collision，主要来自：

```text
stl/right_palm.stl
stl/f_knuckle.stl
stl/f_proximal.stl
stl/f_middle.stl
stl/f_distal_pst.stl
stl/th_proximal.stl
stl/th_middle.stl
stl/th_distal_pst.stl
```

它的 base 到 palm 结构是：

```text
base -> baseX -> baseY -> baseZ -> baseROLL -> basePITCH -> baseHEAVY
baseHEAVY --fixedHEAVY--> wrist
wrist --rh_wrist_joint fixed, rpy=(0,0,pi/2)--> rh_palm
rh_palm -> fingers
```

注意：`shadow_simple` 不是没有任何旋转自由度，它有 6 个虚拟 base DOF；但它没有“手腕相对手掌/前臂”的真实局部 WRJ1/WRJ2。也就是说，它能整体转手，不能在保持前臂/轨迹语义的同时让手掌做更自然的腕部侧抓。

`sr_shadow_hand`：

- URDF: `assets/urdf/sr_grasp_description/urdf/pregrasp_movable_shadowhand.urdf`
- link 数量: 37
- joint 数量: 36
- non-fixed DOF: 30
- fixed joint: 6
- collision: 1 个 forearm cylinder + 19 个 box + 5 个 distal mesh
- 大量 proximal/middle/knuckle collision 是很细的 box，例如：

```text
knuckle:   0.005 0.005 0.005
proximal:  0.008 0.008 0.040
middle:    0.008 0.008 0.015
palm:      0.050 0.020 0.070
```

它的 base 到 palm 结构是：

```text
base -> baseX -> baseY -> baseZ -> baseROLL -> basePITCH -> baseYAW
baseYAW --base_to_forearm fixed--> forearm
forearm --WRJ2 revolute, axis=(0,1,0), origin=(0,-0.010,0.213)--> wrist
wrist --WRJ1 revolute, axis=(1,0,0), origin=(0,0,0.034)--> palm
palm -> fingers
```

### 9.2 两者真正的差别

两者的手指拓扑非常接近：

- FF/MF/RF 基本都是 `J4 -> J3 -> J2 -> J1`
- LF 多一个 `J5`
- TH 是 `J5 -> J4 -> J3 -> J2 -> J1`
- finger joint origins 和 axes 大体同源

主要差别不是手指运动学，而是：

- `shadow_simple` 的 palm/finger collision 更精细，接触更稳定。
- `sr_shadow_hand` 有真实 WRJ2/WRJ1，但 palm/finger collision 过于粗糙。
- `shadow_simple` 的 `wrist -> rh_palm` 有固定 `Rz(90deg)`。
- `sr_shadow_hand` 的 `wrist -> palm` 没有这个固定 90 度，而是通过 WRJ1/WRJ2 和不同 frame 定义表达。
- `shadow_simple` 的动作头是 25：7 个 pose-style action + 18 个 active hand DOF。
- 当前训练里 qpos controller 实际使用 6 个 base action + 18 个 active hand DOF，并通过 passive/mimic 处理 J1。
- `sr_shadow_hand` 的动作头是 30：6 个 base action + 24 个 hand DOF，其中包括 WRJ2/WRJ1。

### 9.3 `ssr_shadow_hand` 可行性判断

可行，而且比修补 sr collision 更合理。

建议做法：

```text
以 shadow_simple URDF 为主体
保留 rh_palm / rh_* fingers / STL collision / passive joints
在 baseHEAVY 和 wrist/rh_palm 之间插入 forearm + WRJ2 + WRJ1
```

一个候选链路结构是：

```text
base -> baseX -> baseY -> baseZ -> baseROLL -> basePITCH -> baseHEAVY
baseHEAVY --fixed--> ssr_forearm
ssr_forearm --WRJ2 revolute--> ssr_wrist
ssr_wrist --WRJ1 revolute--> wrist
wrist --rh_wrist_joint fixed, rpy=(0,0,pi/2)--> rh_palm
rh_palm -> original shadow_simple fingers
```

这样做的好处：

- palm/finger 仍然使用 `shadow_simple` 的 STL collision。
- 原来的 `rh_*` link/joint 名字可以尽量保留，减少训练代码和 reference trajectory 的映射改动。
- WRJ2/WRJ1 作为新增 hand DOF，能表达侧抓所需的腕部姿态。
- 成功率更可能接近 `shadow_simple`，而不是被 sr 粗糙 collision 限制。

### 9.4 需要小心的点

不能直接把 sr 的 `palm` 整段复制到 `shadow_simple`，否则会丢掉 `shadow_simple` 的 STL collision 优势。

不能简单删掉 `shadow_simple` 的 `rh_wrist_joint rpy=(0,0,pi/2)`。这个 fixed transform 已经被它所有 `rh_*` finger joint origin、axis 和 mesh collision 隐含依赖。若删除，需要整体重算 palm 下所有子 link 的 frame，不划算。

更稳妥的做法是保留这个 fixed transform，把 WRJ2/WRJ1 插在它前面。

`ssr_shadow_hand.yaml` 应该是新的 hand 配置，而不是覆盖 `shadow_simple.yaml` 或 `sr_shadow_hand.yaml`。建议配置语义：

```text
name: ssr_shadow_hand
robotAssetFile: shadow_hand_simple/ssr_right_with_wrist.urdf
palm_link: wrist 或 rh_palm，需要根据 reward/obs 语义统一
eef_link: wrist 或 ssr_forearm，需要根据 trajectory 控制语义统一
fingertips_link: ["rh_thdistal", "rh_ffdistal", "rh_mfdistal", "rh_rfdistal", "rh_lfdistal"]
passive_joints: 继续沿用 shadow_simple 的 rh_FFJ1/rh_MFJ1/rh_RFJ1/rh_LFJ1 mimic
num_arm_dofs: 6
handdof: 20
numActions: 27 或 26/30，取决于当前 qpos/pose action 定义
```

动作维度需要单独定：

- 如果沿用当前 qpos 训练范式，最自然是 `6 base + 2 wrist + 18 active fingers = 26`。
- 如果某些 PPO/action 代码仍按 pose-style `7 + hand_dof` 组织，则可能需要 `7 + 20 = 27`。
- 不建议照搬 sr 的 30 维，因为 `shadow_simple` 的 J1 是 passive/mimic，不应该突然变成 4 个独立动作。

### 9.5 后续实现顺序建议

1. 复制 `assets/shadow_hand_simple/right_with_base.urdf` 为 `ssr_right_with_wrist.urdf`。
2. 只修改 base 到 `wrist` 之间的链路，插入 `ssr_forearm`、`ssr_wrist`、`WRJ2`、`WRJ1`。
3. 保留 `wrist -> rh_palm` 的 fixed `Rz(90deg)`。
4. 新建 `tasks/hand/ssr_shadow_hand.yaml`，不要覆盖原两个 hand config。
5. 先用 `play_tra_rot.py` 小规模可视化验证：
   - 0 度时是否和 `shadow_simple` 一致；
   - WRJ2/WRJ1 转动时 palm 是否按预期侧抓；
   - fingertip collision 是否仍然贴合视觉。
6. 再接训练代码的动作维度、checkpoint loading 和 reference action 映射。
